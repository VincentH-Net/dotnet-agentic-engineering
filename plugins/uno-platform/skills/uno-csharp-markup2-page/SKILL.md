# C# Markup 2 Page (View + Model) for Uno Platform

Use when adding a **new page** to a Uno Platform solution that has already been configured for C# Markup 2 (CSharpForMarkup). This is the per-page item template that pairs a markup `.cs` file, a `.logic.cs` code-behind partial, and — for MVVM or MVUX — a viewmodel file.

> This skill covers the user's **CSharpForMarkup / C# Markup 2** library. It is **not** the same technology as the separately-named "C# Markup" from `Uno.Extensions.Markup`. Code shown here is specific to CSharpForMarkup.

## 1. When to use this skill

Select when:

- A Uno 6 solution has already had the Presentation layer added via `mcs-uno-markup2` (see the companion `uno-csharp-markup2-setup` skill) and needs another page added
- The new page should use a specific model type — MVVM, MVUX, or no model — independently of other pages (per-page choice is supported)
- You need to conform to the CSharpForMarkup markup/logic split convention: markup in `<Name>Page.cs`, logic in `<Name>Page.logic.cs`, model in a separate `<Name>Model.cs`

Do not use before the Presentation layer exists — run `mcs-uno-markup2` first, or the generated page will have no project to live in and no supporting `CSharpMarkup.*` packages to reference.

## 2. Prerequisites

- Uno solution with a C# Markup 2 Presentation project (added via `mcs-uno-markup2`)
- `Modern.CSharp.Templates` installed (`dotnet new install Modern.CSharp.Templates`)
- You know the target namespace (typically `<AppRootNamespace>.Presentation` or a sub-namespace)

## 3. Generate the page

Run inside the Presentation project folder (or the desired sub-folder):

```bash
dotnet new mcs-uno-view \
  --name Home \
  --namespace Company.MyApp.Presentation \
  --presentation mvux
```

### Required parameters

| Option | Purpose | Values |
|---|---|---|
| `-n`, `--name` | Page name (becomes `<Name>Page`) | e.g. `Home`, `Details` |
| `-p:n`, `--namespace` | **Full** namespace for the view (NOT a root-namespace prefix — the exact namespace the files will declare) | e.g. `Company.MyApp.Presentation` |
| `-p`, `--presentation` | Model type for the view | `none` / `mvvm` / `mvux` |

### Optional parameters

| Option | Purpose | Values / default |
|---|---|---|
| `-c`, `--csharpversion` | C# language version for generated files | `12` (default) |
| `-o`, `--output` | Output folder | path |

## 4. What each `--presentation` choice generates

### `--presentation none`

- `<Name>Page.cs` — the markup file with the `Build()` method
- `<Name>Page.logic.cs` — partial class for code-behind event handlers and imperative logic

Use for simple pages where a viewmodel adds no value (splash screens, static content, pages driven entirely by navigation).

### `--presentation mvvm`

- `<Name>Page.cs` — markup
- `<Name>Page.logic.cs` — code-behind partial
- `<Name>Model.cs` — viewmodel, typically using `CommunityToolkit.Mvvm` `[ObservableProperty]` / `[RelayCommand]` patterns on `ObservableObject`

Use when the app uses imperative reactive state (property-changed notifications, relay commands).

### `--presentation mvux`

- `<Name>Page.cs` — markup
- `<Name>Page.logic.cs` — code-behind partial
- `<Name>Model.cs` — MVUX model using `Uno.Extensions.Reactive` primitives (`IState<T>`, `IListFeed<T>`, `AsyncCommand`, …)

Use when the app uses immutable-state feeds + commands and you want consistent MVUX across pages.

## 5. Rules every generated page must follow

Generated files already follow these rules — preserve them when editing:

| Rule | File it applies to |
|---|---|
| Use `CSharpMarkup.*` namespaces only — no `Microsoft.UI.Xaml.*`, `Windows.UI.Xaml.*`, `Uno.*` UI-object-model usings | `<Name>Page.cs` |
| No imperative logic, event handlers, or platform-type references | `<Name>Page.cs` |
| Full access to Uno/WinUI types and event handlers; capture controls via `.Assign(out var x)` from the markup side | `<Name>Page.logic.cs` |
| All binding paths expressed as `.Bind(vm.X)` (CallerArgumentExpression), not string literals | `<Name>Page.cs` |
| Attached properties written as `Grid_Row(0)`, `Grid_Column(1)` (type prefix + underscore) | `<Name>Page.cs` |
| Conditional children as `cond ? view : null`; dynamic lists via `Spread(...)` — avoid building trees imperatively | `<Name>Page.cs` |

## 6. Choosing between none / mvvm / mvux

Consistency across the solution matters more than per-page optimization:

- If the Presentation layer was generated with `mcs-uno-markup2 --presentation mvux`, use `--presentation mvux` for new pages unless there is a concrete reason to mix patterns.
- Use `--presentation none` only for pages with no state (splash, about, 404).
- Avoid mixing `mvvm` and `mvux` in the same solution without a clear boundary — the mental models differ (imperative property-changed vs immutable feeds/states).

## 7. Typical placement

Inside the Presentation project, create a folder per feature area and run the template from there so files land alongside their feature:

```bash
mkdir -p Features/Home
dotnet new mcs-uno-view \
  --name Home \
  --namespace Company.MyApp.Presentation.Features.Home \
  --presentation mvux \
  --output Features/Home
```

Resulting structure:

```text
Presentation/
└── Features/
    └── Home/
        ├── HomePage.cs          ← markup (Build())
        ├── HomePage.logic.cs    ← code-behind partial
        └── HomeModel.cs         ← MVUX model
```

## 8. After generation

- Register the page with Uno Extensions Navigation (`views.Register(new ViewMap<HomePage, HomeModel>())`) and add a `RouteMap` entry — see the `uno-hamburgermenu-databinding` skill for the navigation pattern.
- Wire the model into DI if the Presentation layer uses it (MVUX typically does).
